# ZooCute Settings Panel Design
**日期：** 2026-04-01
**状态：** 草案

---

## 目标

为 ZooCute 增加首版设置面板，承接当前 Ribbon 底部已预留的设置入口，并提供以下 3 个可持久化、可立即生效的设置项：

- 主题模式：`跟随系统 / 明亮 / 暗夜`
- 读写模式：`只读 / 读写`
- 插件目录：显示当前目录，并支持 `选择目录`、`恢复默认`、`打开插件目录`

本次设计目标：

- 设置项修改后立即保存，不提供统一的“应用”按钮
- 设置值持久化到本地，下次启动继续生效
- 首版只覆盖用户已经确认的 3 个设置项，不扩展其他偏好项
- 插件目录选择使用系统原生目录选择器，不提供手动输入路径

---

## 非目标

本次不包含：

- 连接相关设置
- 日志、调试、性能类设置
- 语言、字体、布局等外观扩展项
- 插件启用禁用、插件超时、单插件配置
- 云同步或多设备同步
- 设置导入导出

---

## 用户体验

### 入口

- 复用当前 Ribbon 底部的设置按钮作为进入入口
- 点击后打开设置面板
- 首版设置面板可作为应用内覆盖层、抽屉或独立面板呈现，只要不打断当前主界面信息结构即可

推荐首版采用应用内侧边面板或居中弹层，避免引入新的主导航模式。

### 信息结构

设置面板分为 3 个分组：

- 外观
- 安全
- 插件

每个分组只展示当前轮次确认的设置项，不加入占位项。

### 生效方式

- 每个设置项在用户操作完成后立即生效
- 每个设置项在生效的同时立即写入本地持久化存储
- 不提供页面级“保存”“取消”“应用”按钮

---

## 设置项定义

### 主题模式

展示方式：

- 单选项
- 选项为 `跟随系统`、`明亮`、`暗夜`

行为规则：

- 用户切换后立即更新界面主题
- 设置值立即持久化
- 当用户选择 `跟随系统` 时，不直接持久化为 `light` 或 `dark`，而是持久化为“system”语义值
- 应用启动时，如果设置值为 `system`，则根据系统主题决定实际渲染主题

兼容要求：

- 保持当前 `data-theme` 驱动的主题机制
- 在 `system` 模式下继续使用系统主题检测逻辑

### 读写模式

展示方式：

- 单选项
- 选项为 `只读`、`读写`
- 默认值为 `只读`
- 设置项下方展示说明文字：`只读模式下禁止新增、修改、删除节点。`

行为规则：

- 用户切换后立即生效
- 设置值立即持久化
- 当模式为 `只读` 时，前端直接禁用新增、编辑、保存、删除等写操作入口
- 后端命令也需要校验该设置，防止仅靠前端禁用被绕过
- 当模式为 `读写` 时，恢复现有写操作能力

安全原则：

- `只读` 是应用默认模式
- 写操作保护需要覆盖新增、修改、删除三类节点变更动作
- 首版不做更细的权限分级

### 插件目录

展示方式：

- 显示当前生效的插件目录路径
- 提供 3 个按钮：`选择目录`、`恢复默认`、`打开插件目录`

行为规则：

- `选择目录`：
  - 调用系统原生目录选择器
  - 仅允许选择目录，不允许手动输入
  - 选择完成后立即保存并重新扫描插件
- `恢复默认`：
  - 将目录重置为应用默认插件目录
  - 重置后立即保存并重新扫描插件
- `打开插件目录`：
  - 使用系统能力打开当前插件目录
  - 若默认目录尚不存在，可按需先创建再打开

默认目录规则：

- 默认插件目录仍为应用数据目录下的 `plugins`
- Windows 与 macOS 都遵循各自平台的应用数据目录习惯
- 自定义目录是对默认目录的覆盖，不改变默认目录计算规则

首版边界：

- 不支持手动输入路径
- 不支持同时配置多个插件目录
- 不支持“最近目录”历史

## 数据模型

建议新增统一设置对象，至少包含：

```ts
type ThemePreference = "system" | "light" | "dark";
type WriteMode = "readonly" | "readwrite";

interface AppSettings {
  theme: ThemePreference;
  writeMode: WriteMode;
  pluginDirectory: string | null;
}
```

语义说明：

- `theme`
  - `system` 表示跟随系统
  - `light` 表示明亮
  - `dark` 表示暗夜
- `writeMode`
  - `readonly` 表示只读
  - `readwrite` 表示读写
- `pluginDirectory`
  - `null` 表示使用默认插件目录
  - 非空字符串表示使用用户选择的绝对路径

## 持久化方案

首版建议分层处理：

- 前端偏好项：本地持久化
  - `theme`
  - `writeMode`
- 插件目录：由后端参与解析与执行
  - 最终也需要本地持久化，并在启动时由后端读取

设计约束：

- 不让前端与后端分别维护彼此不一致的设置副本
- 应提供单一设置读取入口，避免主题、只读状态、插件目录分别散落在不同 key

推荐方向：

- 统一定义 `AppSettings`
- 启动时加载设置
- 更新任一设置项时整体写回

## 前端设计

### 面板职责

设置面板负责：

- 展示当前设置值
- 响应用户修改
- 触发即时保存
- 触发即时生效

### 主题联动

当前项目已经存在基于 `data-theme` 的主题机制。首版应扩展为：

- `light` 时写入并应用浅色主题
- `dark` 时写入并应用深色主题
- `system` 时根据系统主题实时决定当前 `data-theme`

如果实现成本可控，`system` 模式下应在系统主题变化时同步更新界面主题；若首版暂不监听，也至少保证应用启动时能正确应用。

### 只读联动

前端需要根据 `writeMode` 控制以下能力：

- 隐藏或禁用保存按钮
- 隐藏或禁用新增节点入口
- 隐藏或禁用删除节点入口
- 隐藏或禁用其他会触发写命令的入口

推荐优先使用“禁用并保留入口”的方式，让用户能理解当前能力被只读模式约束。

### 插件目录联动

前端需要：

- 显示当前目录
- 调用后端命令选择目录
- 调用后端命令恢复默认目录
- 调用后端命令打开当前目录
- 当插件目录发生变化后，刷新插件列表

---

## 后端设计

### 设置读取与更新

后端需要能够：

- 提供当前设置读取能力
- 提供单项或整体设置更新能力
- 在应用启动时加载设置

### 写操作保护

后端写命令需要基于 `writeMode` 做最终校验，至少覆盖：

- `save_node`
- `create_node`
- `delete_node`

当设置为 `readonly` 时，这些命令返回明确错误信息。

### 插件目录支持

当前插件根目录是在应用状态初始化时确定的固定值。首版需要改为：

- 默认从应用数据目录推导默认插件目录
- 若设置中存在自定义目录，则优先使用该目录
- 插件目录变化后，后续插件扫描和执行都基于新的生效路径

### 原生能力

插件目录设置需要用到桌面端原生能力：

- 打开系统原生目录选择器
- 打开目录到系统文件管理器

这些行为应由 Tauri 后端提供命令封装，前端只消费结果。

---

## 错误处理

### 主题模式

- 如果设置值损坏或不可识别，回退到 `system`

### 读写模式

- 如果设置值损坏或不可识别，回退到 `readonly`

### 插件目录

- 选择目录取消时，不报错，不修改当前值
- 打开目录失败时，向用户展示错误提示
- 恢复默认后如默认目录不存在，可按需创建
- 自定义目录失效时，不自动静默改写用户设置，但插件列表可为空并给出可理解提示

---

## 测试策略

### 前端测试

- 设置按钮可打开设置面板
- 主题模式切换会更新界面主题并写入持久化
- `system` 模式下会使用系统主题作为初始值
- 读写模式默认值为 `readonly`
- 切换到 `readonly` 后相关写操作入口被禁用
- 插件目录展示当前值
- 点击 `恢复默认` 后目录显示更新

### 后端测试

- 默认设置可正确返回：主题为 `system`，读写模式为 `readonly`
- 更新设置后重启仍可读取到持久化值
- `readonly` 模式下 `save_node`、`create_node`、`delete_node` 会被拒绝
- 自定义插件目录可覆盖默认目录
- 恢复默认后重新回到应用数据目录下的 `plugins`

## 风险与后续扩展

当前风险：

- 主题偏好当前已经有旧的 `localStorage` 存储逻辑，迁移到统一设置结构时需要处理兼容
- 写保护如果只覆盖前端不覆盖后端，会留下绕过风险
- 插件目录切换后需要明确插件列表刷新时机，否则用户会看到旧状态

后续可扩展方向：

- 插件超时设置
- 插件启用禁用管理
- 自动重扫插件目录
- 更多外观设置
- 日志与调试设置

---

## 推荐结论

首版设置页采用最小可用范围：

- 仅实现主题模式、读写模式、插件目录 3 项设置
- 所有设置立即保存、立即生效
- 默认读写模式为 `只读`
- 插件目录使用系统原生目录选择器，不提供手动输入
- 后端承担写保护和插件目录能力，前端负责设置展示与即时反馈

该方案与当前 ZooCute 已有的主题机制、Ribbon 设置入口和插件扫描实现兼容度高，改动边界清晰，也为后续扩展更多设置项保留了统一的数据结构。
