# ZooCute 设计文档

## 1. 目标

ZooCute 是一款面向开发者效率的 ZooKeeper 桌面客户端，强调现代化界面、轻量化体积、跨平台体验，以及日常高频操作的顺手程度。

第一版的核心目标只有两个：

- 更快地定位目标节点
- 更稳地查看和编辑节点数据

产品定位不是企业平台控制台，而是偏 IDE 工作台式的 ZooKeeper 客户端。首版先聚焦个人和小团队的高频使用场景，为后续插件化解码和平台化能力预留接口。

## 2. 用户与场景

目标用户：

- 经常查看 ZooKeeper 配置的开发者
- 需要排查线上配置问题的测试和运维
- 需要在多个集群间切换、比对节点数据的工程师

高频场景：

- 快速进入某个集群并定位指定路径
- 浏览大树结构并查看 children / stat 信息
- 编辑 JSON 或文本配置并安全保存
- 查看不可直接编辑的二进制或序列化内容
- 在多个节点之间切换、对比

## 3. 产品原则

- 识别优先：先判断数据类型，再决定展示和编辑能力
- 安全优先：默认避免误编辑，尤其是未知或序列化内容
- 导航优先：树浏览、搜索必须顺手
- 轻量优先：避免引入过重的运行时和复杂服务依赖
- 跨平台一致：macOS、Windows、Linux 交互保持一致

## 4. 功能范围

### 4.1 v1 包含

- 多连接管理
- 当前连接状态展示与快速切换
- ZooKeeper 真实连接串输入
- 可选 digest 认证
- 节点树懒加载浏览
- 路径搜索与路径直达
- 节点数据查看
- 节点 stat / children 基础信息展示
- 文本和 JSON 编辑
- 保存前 diff
- 未保存提醒
- 版本冲突提示
- 多标签打开节点
- 复制路径、刷新、新建子节点、删除节点等常用操作

### 4.2 v1 不包含

- 团队权限中心
- 审计平台
- 完整 watcher 编排界面
- 批量脚本化修改
- 复杂 ACL 管理工作流
- 依赖企业平台能力的审批和协作功能

## 5. 信息架构

主界面采用三栏 IDE 工作台布局，并将连接上下文从节点树中分离。

### 5.1 顶部全局栏

- 当前连接名称
- 连接状态
- 切换连接入口
- 最近连接入口

连接被视为全局工作区上下文，而不是树导航的一部分。

### 5.2 左侧导航栏

- 搜索框
- 节点树

左侧栏只服务于“快速抵达目标节点”。

### 5.3 中间工作区

- 标签页
- 面包屑路径
- 数据类型状态条
- 内容查看 / 编辑区
- 保存、Diff、原始视图等操作

### 5.4 右侧上下文栏

- stat 信息
- children 摘要
- ACL 摘要
- 快捷操作

在窄屏下可折叠，以保证编辑区优先级。

## 6. 核心交互

### 6.1 导航主流程

1. 用户选择连接
2. 在左侧搜索路径或展开树
3. 打开节点进入中间标签页
4. 查看数据类型与上下文信息
5. 根据可编辑性执行查看、切换视图或保存

### 6.2 编辑主流程

1. 打开节点
2. 系统识别数据类型
3. 如果内容可安全编辑，则进入编辑模式
4. 保存前展示 diff
5. 若版本变化或状态异常，则提醒冲突并阻止盲写

### 6.3 多节点工作流

- 支持多个节点同时打开
- 标签页切换保留滚动和草稿状态
- 允许并排查看作为后续增强方向，但不强制进入 v1

## 7. 数据识别与编辑模型

ZooKeeper 节点数据不应被默认视为“纯文本可编辑”。

### 7.1 数据状态分类

#### A. 文本 / JSON

- 可编辑
- 支持格式化、校验、Diff

#### B. 可识别但不保证可逆的数据

例如 XML、YAML、Base64、URL 编码、常见 KV 文本。

- 默认提供解析视图
- 保存时需提示可能影响原始格式

#### C. 二进制 / 序列化对象

例如 Java 序列化、未知协议对象、压缩数据、缺少 schema 的 Protobuf。

- 默认只读
- 展示十六进制预览、字节长度、编码猜测、摘要

#### D. 未知类型

- 默认只读
- 明确提示当前内容无法安全编辑

### 7.2 用户可见状态标签

中间工作区顶部状态条示例：

- JSON · 可编辑
- 文本 · 可编辑
- Base64 · 谨慎编辑
- 二进制 · 只读
- 未知类型 · 只读

### 7.3 强制编辑策略

首版采用“默认安全，只读优先”的策略，同时提供受控的高级能力：

- 默认只读
- 用户可主动切换到原始视图
- 对不可安全解析的节点，允许进入“强制原始编辑”模式
- 强制编辑需明确提醒风险

## 8. 插件化解码能力

首版不要求完整插件市场，但需要预留解码器扩展机制。

目标：

- 后续支持 Java 序列化识别
- 支持 Protobuf / Hessian / 自定义协议解码
- 将“是否可编辑”和“如何展示”交由解码器协商决定

建议设计一个统一的数据解释接口，让 UI 只依赖解释结果，而不依赖具体协议细节。

## 9. 技术架构

### 9.1 总体方案

- 桌面壳：Tauri
- 前端：现代前端框架，优先 React 或 Vue
- 后端核心：Rust

### 9.2 ZooKeeper 访问层

建议采用：

- Tauri UI
- Rust `zk-core`
- 可替换的 ZooKeeper client adapter

不把某一个 Rust ZooKeeper client 直接耦合进 UI 层，而是在其外封装稳定接口。

### 9.2.1 最小真实接入阶段

在 MVP 原型之后，优先接入一个“只读 + digest 认证”的最小真实 ZooKeeper 访问层。

这一阶段只支持：

- `connect(connection_string, username?, password?)`
- `list_children(path)`
- `get_node(path)`

这一阶段明确不支持：

- 写节点数据
- create / delete
- watcher
- ACL 编辑
- 多连接持久化
- 自动重连优化

连接串输入格式以 ZooKeeper 常见连接串为准，例如：

- `127.0.0.1:2181`
- `zk1:2181,zk2:2181,zk3:2181`

如果用户填写用户名和密码，则后端在会话建立后执行 digest 认证；如果为空，则按匿名方式连接。

前端顶部连接区最小包含：

- Connection String
- Username
- Password
- Connect 按钮

连接结果至少要能区分：

- 连接失败
- 认证失败
- 路径不存在
- 节点读取失败

前端连接状态建议至少保留：

- `connected`
- `authMode: anonymous | digest`
- `authSucceeded`

### 9.3 `zk-core` 建议职责

- 连接与断开
- 会话状态管理
- 读取节点数据
- 写入节点数据
- 列出 children
- create / delete
- stat / acl 查询
- watch subscribe / cancel
- 数据类型识别与解释器调度

### 9.4 为什么要做适配层

- Rust ZooKeeper client 生态可能不如 Java 成熟
- 未来如果要替换底层实现，不应影响 UI 与交互模型
- 可以先用 Rust client 做 v1 验证，再根据稳定性扩展

## 10. 非功能要求

- 启动速度快
- 包体尽量小
- 离线可用，不依赖远程控制面
- 在大树节点场景中仍保持可接受的导航响应速度
- 崩溃或断连时不丢失未保存草稿

## 11. 风险与对策

### 11.1 数据格式误判

风险：把不可安全编辑的数据当成可编辑内容。

对策：

- 识别优先
- 默认保守
- 强制编辑需明确二次确认

### 11.2 Rust client 兼容性不足

风险：重连、watch、认证等语义不稳定。

对策：

- 通过 `zk-core` 抽象隔离
- 首版优先验证连接稳定性、读写兼容性、watch 语义

### 11.3 UI 过于拥挤

风险：三栏布局容易回到老式运维工具的沉重感。

对策：

- 顶部全局栏与左侧导航分层
- 右侧栏可折叠
- 视觉上强调留白和主要任务区

## 12. 后续演进方向

- 解码器插件系统
- Watcher 视图增强
- 更完整的 ACL 管理
- 节点内容对比视图增强
- 多集群并行查看
- 协作与审计能力

## 13. 当前结论

ZooCute 第一版应以“IDE 工作台式 ZooKeeper 客户端”为核心方向，采用顶部连接栏、左侧导航树、中间内容工作区、右侧上下文栏的主布局，重点打磨节点导航和数据编辑体验，并将不可安全编辑的数据默认设为只读，同时为后续协议解码和底层 client 替换预留扩展空间。
