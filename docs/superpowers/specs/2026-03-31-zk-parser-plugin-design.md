# ZooCute ZooKeeper Parser Plugin Design
**日期：** 2026-03-31
**状态：** 草案

---

## 目标

为 ZooCute 增加一个“插件解析”能力，让用户可以对 ZooKeeper 节点中的二进制数据执行外部解析器，并在当前内容区域直接查看解析结果。

本次设计目标：

- 让用户在节点内容工具栏中手动选择一个解析插件
- 通过外部命令执行插件，不把解析逻辑硬编码进应用
- 将当前节点的原始二进制数据通过 `stdin` 传给插件
- 将插件的 `stdout` 作为纯文本结果展示在前端
- 解析成功后切换到“插件结果”视图
- 解析失败时保持当前视图不变，仅提示错误
- 第一版从固定插件目录自动发现插件，并为后续“手动配置插件”预留结构

---

## 非目标

本次不包含：

- 内嵌 JVM、Python 运行时或任意语言解释器
- 自动根据节点路径或内容类型匹配插件
- 插件返回富文本、HTML 或复杂结构化 UI
- 插件结果编辑、保存回 ZooKeeper
- 完整的插件设置页或手工录入插件命令 UI
- 多插件串联执行
- 沙箱隔离、权限白名单、签名校验等高级安全能力

## 用户体验

### 工具栏

在现有 `RAW / JSON / XML` 工具栏区域中新增：

- 插件下拉框
- `Parse` 按钮
- `PLUGIN` 视图 tab

交互顺序：

1. 用户打开一个节点
2. 如果系统发现可用插件，工具栏展示插件下拉框
3. 用户手动选择一个插件
4. 点击 `Parse`
5. 后端执行插件
6. 成功时切换到 `PLUGIN` 视图并展示纯文本结果
7. 失败时停留在当前视图，仅弹出错误提示

### 内容展示

- `PLUGIN` 视图直接复用当前节点内容主区域
- 不新增独立的右侧结果面板或上下分栏
- 插件结果只读展示，按纯文本渲染
- 用户可随时切回 `RAW / JSON / XML`

### 节点切换

- 切换节点时清空该节点以外的插件结果显示状态
- 默认回到常规视图，不自动继承上一个节点的 `PLUGIN` 结果

## 功能边界

### 发现方式

第一版使用“固定目录自动发现”：

- 应用启动时或首次需要时扫描固定插件目录
- 每个插件目录包含一个定义文件
- 前端仅展示扫描成功且启用的插件

后续可扩展：

- 在设置页中手动配置插件目录或命令
- 支持禁用/启用插件
- 支持自定义排序

### 输入数据

第一版插件只接收一个输入：

- 当前节点的原始二进制字节

明确不传：

- 节点路径
- ZooKeeper Stat 元信息
- 连接信息
- 已解码文本

### 输出数据

第一版只支持一种输出：

- 插件 `stdout` 输出的纯文本

明确不支持：

- JSON schema 驱动渲染
- HTML 渲染
- 图片、二进制附件

## 插件协议

### 目录结构

建议约定如下：

```text
plugins/
  dubbo-provider/
    plugin.json
    parser.jar
```

### 定义文件

第一版插件定义文件建议为 `plugin.json`，最小字段：

```json
{
  "id": "dubbo-provider",
  "name": "Dubbo Provider Decoder",
  "enabled": true,
  "command": "java",
  "args": ["-jar", "parser.jar"]
}
```

字段说明：

- `id`：稳定唯一标识，用于前后端调用
- `name`：前端展示名称
- `enabled`：是否启用
- `command`：启动命令
- `args`：命令参数数组

### 执行方式

后端执行插件时：

1. 根据 `id` 找到插件定义
2. 在插件目录作为工作目录启动外部命令
3. 将当前节点原始字节写入插件进程 `stdin`
4. 读取 `stdout`
5. 读取 `stderr` 和退出码
6. 成功时返回 `stdout` 文本
7. 失败时将错误包装为前端可展示的失败信息

### Java 插件示例

例如：

```json
{
  "id": "hessian-decoder",
  "name": "Hessian Decoder",
  "enabled": true,
  "command": "java",
  "args": ["-jar", "hessian-decoder.jar"]
}
```

插件代码从 `System.in` 读取原始字节，从 `stdout` 输出解析结果文本。

## 前端设计

### 状态模型

建议新增以下状态：

- 已发现插件列表
- 当前选中的插件 `selectedPluginId`
- 当前节点插件结果 `pluginResult`
- 当前是否正在执行插件 `isPluginParsing`
- 当前展示模式中新增 `plugin`

建议 `pluginResult` 至少包含：

- `pluginId`
- `pluginName`
- `content`
- `generatedAt`

### 组件拆分

#### `EditorToolbar`

负责：

- 展示插件下拉框
- 展示 `Parse` 按钮
- 展示 `PLUGIN` tab
- 根据状态控制按钮可用性

交互规则：

- 无插件时，下拉框为空且 `Parse` 禁用
- 未选择插件时，`Parse` 禁用或点击后直接提示
- 插件解析执行中，按钮进入 loading/disabled 状态
- 当当前节点已有插件结果时，`PLUGIN` tab 可点击
- 当没有插件结果时，`PLUGIN` tab 不可点击

#### `EditorPanel`

负责：

- 管理当前视图模式
- 管理当前选中插件
- 触发插件解析命令
- 在节点切换时清理当前插件结果

#### `NodeContentPanel`

新增 `plugin` 分支：

- 当视图模式为 `plugin` 时，按只读纯文本展示插件输出
- 不参与编辑态
- 不允许保存

### 视图规则

建议将现有 `ViewMode` 从：

- `raw | json | xml`

扩展为：

- `raw | json | xml | plugin`

但语义上保留区分：

- `raw/json/xml` 是内置视图
- `plugin` 是外部解析结果视图

不要把“每个插件”做成一个新的视图枚举值，避免状态爆炸。

## 后端设计

### 新命令

新增两个 Tauri 命令：

#### `list_parser_plugins`

职责：

- 扫描固定插件目录
- 读取并校验 `plugin.json`
- 返回可展示插件列表

返回字段建议包含：

- `id`
- `name`
- `enabled`

#### `run_parser_plugin`

输入：

- `pluginId`
- 当前节点原始字节

职责：

- 解析插件定义
- 启动进程
- 将原始字节写入 `stdin`
- 收集 `stdout/stderr`
- 返回成功文本或结构化错误

### 原始字节来源

这是第一版的关键基础能力。

当前节点详情不仅要有展示用的 `string` 值，还需要保留原始字节，供插件执行时使用。后端不能只基于已解码文本反推字节，否则会破坏二进制数据的准确性。

因此需要保证：

- 读取 ZooKeeper 节点时可以拿到原始 `Vec<u8>`
- 插件执行路径直接消费原始字节
- 文本展示和插件执行共享同一份原始数据来源

实现上可以是：

- 将原始字节存入节点详情缓存
- 或新增单独的读取原始字节命令

推荐优先复用节点详情/缓存路径，避免为同一节点重复发起读取。

## 错误处理

### 插件发现失败

- 不阻塞节点浏览
- 插件区域显示为空状态
- `Parse` 按钮禁用

### 用户未选择插件

- 不调用后端
- 前端直接提示“请先选择插件”

### 插件执行失败

失败包括：

- 命令不存在
- 启动失败
- 非零退出码
- `stdout` 读取失败
- 超时

前端行为：

- 保持当前内容视图不切换
- 不覆盖现有内容区
- 弹出错误提示

错误提示建议至少包含：

- 插件名称
- 简要失败原因
- 如可用则附带退出码

### 插件输出为空

- 视为成功
- 切换到 `PLUGIN` 视图
- 展示空文本结果

## 测试策略

### 前端测试

- `EditorToolbar` 显示插件下拉框、`Parse` 按钮、`PLUGIN` tab
- 没有插件结果时 `PLUGIN` tab 不可用
- 解析成功后切换到 `PLUGIN` 视图
- 解析失败后不切换视图
- 节点切换后插件结果被清空

### 前端状态测试

- 选中插件后状态正确持有
- 节点切换时视图从 `plugin` 回退到默认内置视图
- 已存在插件结果时切换 `RAW/JSON/XML/PLUGIN` 正常

### Rust 后端测试

- 插件目录扫描成功
- 非法 `plugin.json` 被跳过或报错
- 外部命令可以收到 `stdin` 原始字节
- `stdout` 成功返回
- `stderr` 和非零退出码被正确转换为失败
- 缺失命令、超时、空输出场景覆盖

## 风险与后续扩展

### 当前风险

- 外部命令执行天然带来安全风险，第一版仅作为本地受信任插件能力
- 原始字节缓存如果处理不当，会增加内存占用
- 插件执行时间过长会影响交互体验，需要超时控制

### 后续扩展方向

- 手动配置插件
- 插件目录设置页
- 自动匹配插件
- 向插件传递节点路径/Stat 元数据
- 支持插件声明输出类型，如 `text/json`
- 支持插件帮助文档和测试运行

---

## 推荐结论

第一版采用以下方案最稳妥：

- 固定目录自动发现插件
- `plugin.json` 描述插件命令
- 后端以插件目录为工作目录执行外部命令
- 原始二进制通过 `stdin` 传入
- `stdout` 作为纯文本结果
- 前端将结果放入 `PLUGIN` 视图并覆盖当前内容展示区
- 执行失败时保留当前视图，仅弹出错误提示

这个方案与当前 ZooCute 的编辑器结构兼容度高，改动边界清晰，也为后续扩展“手动配置插件”和更丰富的插件协议保留了空间。
