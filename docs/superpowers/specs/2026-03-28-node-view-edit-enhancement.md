# ZooCute 节点查看与编辑增强规格

**日期：** 2026-03-28
**状态：** 草案

---

## 目标

针对当前节点查看、编辑、保存体验中“信息区过空、编辑权限边界不清、内容区层次弱、格式化能力不足、树交互偏生硬”等问题，定义一轮专项优化方案。

本次文档聚焦以下目标：

- 优化右侧节点详情区的布局密度与视觉层次
- 重构编辑权限模型，默认只读，显式开启编辑
- 增强节点内容展示能力，支持多视图与编码选择
- 明确二进制 / 序列化数据的展示边界与后续扩展方向
- 为树展开收起补充轻量动效，提升操作反馈

---

## 现状问题

### 1. 右侧内容区视觉上偏空

当前右侧区域虽然已经拆成路径头部、Stat 区、编辑区，但内容块之间的密度不足，主内容区域没有形成明确“工作焦点”，导致：

- 页面中部留白偏多
- 查看模式与编辑模式的差异不够明显
- 用户不容易一眼找到“当前正在看哪里、改哪里”

### 2. 编辑权限模型不够稳

当前文本节点和 JSON 节点在打开后即可直接编辑，这与“默认安全，只读优先”的产品原则不一致，容易造成：

- 用户在仅想查看时误改内容
- 节点切换后无法快速区分“浏览态”与“编辑态”
- 保存前心理负担偏高

### 3. 节点内容区缺乏背景和层次

当前 textarea 与外层容器的视觉差异较弱，节点内容主体没有被明确框出，导致：

- 大段内容的阅读边界不清晰
- 与上方 Stat 区、路径区缺少结构分隔
- 在浅色主题下尤其不容易识别主编辑区域

### 4. 展示格式能力偏弱

当前节点内容基本只有原始文本编辑区，缺少：

- RAW / JSON / XML 等视图切换
- 编码方式切换
- 解析失败时的清晰反馈
- 对二进制、序列化内容的更强解释能力

### 5. 树交互反馈偏硬

当前树的展开与收起是瞬时切换，缺少轻量过渡，容易显得：

- 点击后“跳变”
- 层级结构变化突兀
- 在频繁浏览时缺少顺滑感

## 一、右侧节点详情区布局优化

### 设计目标

让右侧区域从“松散摆放的信息块”变成“清晰分区的工作台”。

### 布局建议

右侧改为四段式结构：

1. 节点路径与状态头部
2. 元数据摘要区
3. 内容工具栏
4. 内容主面板

建议结构：

```text
┌──────────────────────────────────────────────┐
│ 路径 / 标签 / 未保存状态 / 编辑开关          │
├──────────────────────────────────────────────┤
│ Stat 卡片区                                  │
├──────────────────────────────────────────────┤
│ 视图模式 / 编码 / Diff / 放弃 / 保存         │
├──────────────────────────────────────────────┤
│ 节点内容主面板                               │
│                                              │
│ 代码视图 / 格式化视图 / 错误提示 / 空状态     │
│                                              │
└──────────────────────────────────────────────┘
```

### 布局优化原则

- 控制主内容最大宽度，不要无限拉伸
- 使用卡片化容器，让内容主面板成为视觉焦点
- 路径头部与工具栏紧凑化，减少无意义空白
- Stat 区改为更紧凑的卡片栅格，而非松散铺满
- 主内容面板占据更多垂直空间，减少页面“漂浮感”

### 建议视觉处理

- 右侧内容整体增加内边距层级
- Header、Stat、Toolbar、Content 使用不同但接近的背景层次
- 内容面板使用更明显的边框和内阴影或 inset 背景
- 在大屏下限制阅读宽度，避免一整行过长

## 二、编辑交互重构

### 核心原则

默认只能查看，必须显式进入编辑态后才允许修改。

### 新交互模型

节点打开后默认进入”查看态”：

- 内容可选择、可复制
- 内容不可修改
- 保存、Diff、放弃修改按钮隐藏
- Header 行显示”开启编辑”开关

用户点击”开启编辑”后进入”编辑态”：

- 内容区域变为可编辑
- Header 行显示编辑态标识，”开启编辑”开关切换为激活状态
- 可触发草稿变更、Diff、放弃修改、保存

用户点击”放弃修改”或切换节点时：

- 若无未保存修改，直接回到查看态
- 若有未保存修改，弹出确认
- 确认放弃后退出编辑态

### 工具栏控件状态表

**Header 行：**

| 控件 | 查看态 | 编辑态·无草稿 | 编辑态·有草稿 |
|---|---|---|---|
| 路径 / 标签 | 显示 | 显示 | 显示 |
| 开启编辑开关 | 显示·未激活 | 显示·已激活 | 显示·已激活 |
| 未保存状态标记 | 隐藏 | 隐藏 | 显示 |

**工具栏：**

| 控件 | 查看态 | 编辑态·无草稿 | 编辑态·有草稿 |
|---|---|---|---|
| 视图模式（Raw / JSON / XML） | 可切换 | 可切换 | 可切换 |
| 字符编码（Charset） | 可切换（文本节点） | 可切换（文本节点） | 可切换（文本节点） |
| 查看 Diff | 隐藏 | 禁用 | 可点击 |
| 放弃修改 | 隐藏 | 显示·直接退出编辑态 | 显示·弹确认后退出 |
| 保存 | 隐藏 | 禁用 | 可点击 |

"放弃修改"兼做"退出编辑"：无草稿时点击直接退出，无需确认；有草稿时弹确认对话框。

### Diff 定义

Diff 对比的是**当前草稿**与**服务端当前值**（实时从 ZooKeeper 读取，不是上次本地保存的快照）。

- 仅在编辑态且有草稿变更时可用
- 点击后拉取服务端最新值，与当前草稿做文本 Diff 展示
- 若拉取失败，提示"无法获取服务端当前值"，不展示 Diff
- Diff 展示为只读，不允许在 Diff 视图中直接编辑

### 节点类型与权限规则映射

| 节点内容特征 | 权限规则 | 判断依据 |
|---|---|---|
| 合法 UTF-8 文本（含空节点） | 可安全编辑 | 可无损往返 |
| 可识别为 JSON / XML 的格式化内容 | 谨慎编辑 | 保存时可能被规范化，格式可能变化 |
| GBK / ISO-8859-1 等非 UTF-8 文本 | 谨慎编辑 | 编码转换可能引入字节变化 |
| 二进制 / 序列化内容（Kryo、Java Serializable 等） | 只读 | 无法保证安全回写 |

判断顺序：先尝试解析为 UTF-8 文本，若成功再判断是否为 JSON/XML；若非 UTF-8，尝试常见字符集；否则归为二进制只读。

### 权限规则

#### 对可安全编辑内容

- 默认查看态
- 用户手动开启编辑后才能修改

#### 对谨慎编辑内容

- 默认查看态
- 开启编辑时额外提示”可能改变原始格式”

#### 对只读内容

- 默认查看态
- 不显示普通”开启编辑”
- 仅在高级操作区显示”强制原始编辑”
- 强制原始编辑需要二次确认

### 保存后的状态规则

保存成功后应当：

- 清除草稿态
- 将当前展示基线同步为最新保存值
- 若后端返回标准化结果，以后端返回内容为准
- Diff 基线同步更新，避免继续拿旧值比较
- **自动退回查看态**

退回查看态的理由：保存 = 操作完成 = 回归安全态，与”默认只读”原则一致。若用户需要继续修改，主动再次开启编辑，成本极低。

## 三、节点内容区视觉增强

### 目标

让用户一眼识别“这块是节点内容正文”。

### 建议样式

- 内容区外层为独立卡片容器
- 使用明显区别于页面背景的 `panel-inset` 背景
- 增加 1px 边框和轻微圆角
- 内容区顶部可加标题行，例如“节点内容”
- 文本区使用代码风格背景，而不是直接裸露在页面上

### 建议层级

- 页面背景：`bg-canvas`
- 右侧区块背景：`bg-subtle`
- 内容主面板：`bg-panel` 或 `bg-elevated`
- 实际文本区：`bg-inset`

### 用户感知收益

- 更容易聚焦正文
- 更容易区分“头部信息”和“实际内容”
- 在查看长文本、JSON、XML 时层次更清楚

## 四、视图模式与编码能力增强

### 目标

让节点内容不仅能“看原始字符串”，还能按可识别格式进行更友好的展示。

### 视图模式

建议在内容工具栏提供以下视图模式：

- `Raw`
- `JSON`
- `XML`

后续可扩展但本次不强制：

- `Text`
- `Hex`
- `Decoded`

### 视图模式行为

#### Raw

- 始终可用
- 展示节点原始文本或原始字节转码结果
- 是所有保存操作的最终基准视图

#### JSON

- 尝试将当前内容解析为 JSON
- 成功则格式化展示
- 失败则提示“当前内容无法转换为 JSON”
- 在失败状态下不可伪装成 JSON 编辑器

#### XML

- 尝试将当前内容解析为 XML
- 成功则格式化展示
- 失败则提示“当前内容无法转换为 XML”

### 字符编码选择（Charset）

工具栏提供字符编码下拉，首版支持：

- `UTF-8`（默认）
- `GBK`
- `ISO-8859-1`

说明：

- Charset 仅影响 Raw / JSON / XML 视图中文本内容的展示
- 不影响节点原始字节，不参与保存逻辑
- **仅对文本内容节点开放**；二进制 / 序列化内容节点不显示 Charset 控件
- 若按所选编码解码失败，显示明确错误提示，不自动替换或吞错

### 解析失败提示规则

解析失败时，应显示内联错误区，而不是静默回退：

- JSON 失败：`转换失败：当前内容不是合法 JSON`
- XML 失败：`转换失败：当前内容不是合法 XML`
- 编码失败：`转换失败：无法按所选编码解码当前字节序列`

错误提示区需要包含：

- 失败原因
- 当前视图模式
- 当前编码
- 返回 Raw 的快捷操作

### 编辑权限与视图模式

**只允许对文本内容节点开启编辑**，二进制 / 序列化节点仅支持查看（含强制原始编辑路径）。

视图模式与编辑态的交叉行为：

| 视图模式 | 编辑态，解析成功 | 编辑态，解析失败 | 查看态 |
|---|---|---|---|
| Raw | 直接编辑文本 | — | 只读 |
| JSON | 在格式化视图内编辑，保存时序列化回文本 | 自动降级到 Raw；显示解析错误；不可保存 | 只读格式化展示 |
| XML | 在格式化视图内编辑，保存时序列化回文本 | 自动降级到 Raw；显示解析错误；不可保存 | 只读格式化展示 |

**保存采用所见即所得原则**：JSON/XML 视图编辑后的内容，以当前格式化视图中的字符串原样写回，不做额外压缩或 pretty-print 转换。用户在 JSON 视图看到什么格式，保存到 ZooKeeper 的就是什么格式。

### 保存失败行为

保存失败时：

- **停留在编辑态**，草稿内容完整保留
- 在内容区顶部显示内联错误提示（含失败原因）
- 用户可修改后重试，或放弃修改退出编辑态

## 五、关于 PrettyZoo 能展示 Kryo 类名和字段名的原因

### 结论

PrettyZoo 之所以有时能展示部分“类名”和“参数名”，并不代表它完整理解了业务对象；更可能是它对某些常见二进制 / 序列化格式做了启发式识别或局部解码。

### 可能原因

#### 1. 序列化数据本身就带有类名信息

很多序列化协议为了支持反序列化，会在字节流中保留：

- 全限定类名
- 字段名
- schema 标识
- 类型标签

如果这些内容以字符串形式混在字节流中，工具即使不完全反序列化，也可能扫描出可读片段。

#### 2. Kryo 默认会写入类注册信息或类名

Kryo 在没有完全注册优化、或序列化策略未极致压缩时，字节流中可能包含：

- 类名
- 集合元素类型
- 字段相关元数据

这使得工具有机会“半解析”出类名或部分字段含义。

#### 3. 工具可能做了启发式字符串提取

一些 ZooKeeper 客户端不会真正把对象还原成强类型实例，而是：

- 扫描可打印字符串
- 猜测字节流中的 UTF-8 片段
- 把疑似类名、字段名、枚举值展示出来

这种能力更接近“结构化预览”，不是可靠反序列化。

#### 4. 工具内置了特定协议识别逻辑

PrettyZoo 可能对常见格式做了专门判断，例如：

- Java Serializable
- JSON / XML / 文本
- 常见压缩流
- 部分二进制对象头特征

如果命中了规则，就能展示比纯 Hex 更多的信息。

### 对 ZooCute 的意义

ZooCute 不应在没有把握的情况下宣称“已正确解析 Kryo”，而应采用分层策略：

- 第一层：识别是否为文本、JSON、XML、常见编码
- 第二层：尝试提取可读字符串片段
- 第三层：显示“疑似类名 / 字段名 / 可打印片段”
- 第四层：未来再通过插件化解码器支持真正的 Kryo 解释

### 本次建议

本次先不承诺完整 Kryo 解析，只建议新增：

- 可打印字符串片段预览
- 疑似类名提示
- 字节长度、编码猜测、十六进制预览
- “当前内容可能包含序列化对象，暂不支持可靠编辑”的文案

## 六、树展开收起动画

### 目标

在不引入厚重动画库的前提下，让树的展开收起更自然。

### 建议方案

- 展开箭头增加 120ms - 180ms 的旋转过渡
- 子节点列表增加轻量高度 / opacity 过渡
- 动画节奏以“快进快出”为主，不拖沓

### 动效原则

- 动画必须短
- 不能影响大树浏览性能
- 不能因测量高度造成明显卡顿
- 对频繁展开收起操作保持稳定

### 推荐表现

- 箭头：`rotate(0deg)` 到 `rotate(90deg)` 或视觉等效角度
- 子树：使用 CSS Grid 折叠方案，无需 JS 测量高度：
  - 收起：`grid-template-rows: 1fr → 0fr`，子内容设 `overflow: hidden`
  - 展开：`grid-template-rows: 0fr → 1fr`
  - 优点：动画节奏自然，不依赖 `max-height` hack，无测量卡顿
- 收起时快速淡出，不做复杂回弹

### 无障碍要求

- 在 `prefers-reduced-motion: reduce` 下关闭或显著减弱动画

## 九、推荐结论

如果本轮目标是“先把节点查看和编辑做稳”，建议优先级如下：

1. 先做“默认查看、显式开启编辑”
2. 再做右侧布局和内容区视觉增强
3. 然后补视图模式与编码切换
4. 最后再做二进制增强预览和树动画

这样可以先解决误编辑和状态混乱问题，再继续提升高级能力与细节体验。
